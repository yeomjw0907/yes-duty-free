대상: 대표님 공유용

## 1) 한눈에 보는 현재 상태

- 프론트/백엔드 개발 구현: **완료(1차)**
- 인프라 배포(웹 + Edge Function): **완료**
- 실제 결제 검증(E2E): **부분 완료**
- 상용 오픈 준비: **포트원 콘솔 설정값(키/웹훅) 입력 대기**

요약하면, 개발은 거의 끝났고,

## 2) 무엇을 구현했는지

이번 작업에서 결제 흐름을 아래처럼 바꿨습니다.

1. 사용자가 결제 버튼 클릭
2. 주문 생성
3. 서버가 PortOne 결제 링크 생성
4. 고객이 PortOne 결제창에서 결제
5. PortOne 웹훅으로 결제 결과 수신
6. 주문 상태(`결제완료/실패/환불`) 자동 동기화
7. 고객이 복귀했을 때 주문번호 기준으로 결제 상태 표시

### 주요 구현 항목

- 체크아웃에서 PortOne 결제 링크로 이동하도록 변경
- 결제 복귀 후 주문번호 기반 상태 조회/표시 기능 추가
- 주문 상세 화면에 `결제 상태`, `결제 일시` 표시 추가
- PortOne 웹훅 수신 및 서명 검증 로직 구현
- 결제 결과에 따라 `orders.payment_status`, `paid_at`, `orders.status` 자동 업데이트

### 2-1) 고객 여정 기준 결제 프로세스

고객 입장에서 실제 흐름은 아래와 같습니다.

1. 회원가입/로그인
2. 상품 선택 -> 장바구니 -> 배송지 입력
3. 결제 버튼 클릭
4. PortOne 결제창으로 이동
5. 카드/간편결제 등으로 결제
6. 결제 완료 후 쇼핑몰로 복귀
7. 주문완료 화면에서 결제 상태 확인
8. 결제 완료 주문만 배송 준비/출고 프로세스로 이동

운영팀 관점:
- 고객 문의 시 주문번호로 `결제 상태`, `결제 시각`, `주문 상태`를 동시에 확인 가능

### 2-2) 돈의 흐름(정산 흐름) 설명

비개발 관점에서 가장 중요한 부분만 간단히 정리하면:

- PortOne은 결제 오케스트레이션/중계 역할
- 실제 승인/매입/정산은 연결된 PG/결제수단 사업자 정책에 따름
- 즉, 돈은 PortOne에 쌓이는 구조가 아니라, PG/가맹점 정산 구조로 흐름

실무 흐름:
1. 고객 결제 승인
2. 결제 결과가 PortOne -> 우리 서버(웹훅)로 전달
3. 우리 주문 상태가 결제완료로 갱신
4. 이후 정산일정(T+N 등)에 따라 PG가 가맹점 정산계좌로 입금

현재 단계 안내:
- 지금은 샌드박스 중심 검증 단계라 실제 입금은 발생하지 않음
- 라이브 전환 시 PG 계약 조건(수수료, 정산주기, 공휴일 반영)에 따라 실제 입금 스케줄이 확정됨

정산 관련 운영팀 확인 포인트:
- 어떤 PG 채널로 결제했는지
- 채널별 수수료율/정산주기
- 정산계좌 등록/검증 완료 여부
- 환불 발생 시 차감 반영 규칙

## 3) 번역/운영/배송/환율 처리 상세

비개발 직군에서도 바로 이해할 수 있도록, 실제 서비스에서 어떻게 동작하는지 항목별로 정리했습니다.

### 3-1) 번역(다국어) 처리

- 현재 서비스는 다국어 구조를 이미 사용 중이며, 이번 결제 연동은 **기존 번역 구조를 깨지 않도록** 구현했습니다.
- 결제 성공/실패/대기 표시는 우선 한국어 문구로 적용했고, 기존 i18n 확장 구조를 유지해서 추후 영문/중문 확장이 가능합니다.
- 운영 관점 의미:
  - 지금은 기능 안정성(결제 동기화)이 우선
  - 문구 다국어 확장은 추가 리소스 투입 시 빠르게 확장 가능

### 3-2) 운영(백오피스/CS) 처리

- 주문 상세에서 결제 상태를 바로 확인 가능하도록 변경:
  - `payment_status` (pending/paid/failed/refunded)
  - `paid_at` (실제 결제 완료 시각)
- 웹훅 수신 시 주문 상태를 자동 반영하도록 구성:
  - 결제 성공 시 `orders.status`를 `상품준비중`으로 자동 전환
- 운영 관점 효과:
  - CS팀이 “결제됐는지/언제 됐는지”를 바로 확인 가능
  - 수동 확인/개발팀 문의 빈도 감소

### 3-3) 배송 처리

- 배송 프로세스는 기존 주문 상태 체계를 그대로 유지합니다.
- 결제와 배송의 연결 규칙만 명확히 추가:
  - 결제 성공 -> 배송 준비 단계(`상품준비중`)로 자동 전환
  - 결제 실패 -> 배송 상태는 진행되지 않음
- 운영 관점 효과:
  - 결제 완료 주문만 정상적으로 출고 프로세스에 진입
  - 오출고/오처리 리스크 감소

### 3-4) 환율 처리 (KRW/TWD)

- 기준 정책:
  - 배송국가가 `TW`이면 TWD 결제 우선
  - 그 외 국가는 KRW 결제
- 계산 방식:
  - 주문 원화 금액을 `exchange_rates`의 TWD 레이트로 환산하여 결제 요청 금액 계산
- 안전장치(fallback):
  - TWD 환율이 없거나 비정상이면 KRW 결제로 fallback
- 운영 관점 의미:
  - 환율 데이터 공백 시 결제 중단 대신 서비스 연속성 확보
  - 다만 TWD 정밀 운영을 위해 환율 동기화 작업(자동/수동) 필요

## 5) 테스트 현황
